Introduction

Many in the Geographic Information Systems (GIS) world do not check the reliability of their data before plotting it out on a map, submitting it, and then calling it a day. As many readily accept data ripe and ready for mapping, certain sets, intentionally or unintentionally, may carry some mistakes, high variability, and other overlooked elements.

Before data is mapped out in programs such as ArcPro and QGIS, the underlying functions of the data should be examined, cleaned up, and evaluated for optimal geographic representations and statistical analysis. This also strengthens educational experience in the R software and how it can be prepared as a tool in conjunction with the mapping programs.

This vignette in particular walks through a dataset pertaining to race populations and other demographics in New York City, New York among its respective census tracts.

Examine Data for Quality

After setting up the data, the path to the “cb_2016_us_tract510_500k_nyc_wstats.gpkg” geopackage has been located and its contents will be the focus for the remainder of the script. At this point, the data layers and attributes within that geopackage can examined for readiness. To verify the layers, the function “st_layers” will list the available layers within the geopackage. These are spatial layers that contain the features geographically referenced in the map visualization.

While the layers have now been checked, there is one last step deeper that must be taken for a thorough quality check. The innermost chunk of the data, the attributes within each layer, must be examined. The attributes of each layer provide context and test for both quality control and analytical capability. For example, if we attempted to do a nearest neighbor calculation without looking at the attributes to understand that this data is a multipolygon feature, it would confuse the GIS program and start calculating a loop of incorrect outputs.

For this script, we would like to pull New York City census tract data. This first requires importing the “nyc_tract” data from a specific layer. The only layer pulled that contains the data is “cb_2016_us_tract510_500k_nyc5” within the geopackage by the function “st_read”. The <- symbol is known as an assignment operator, which takes the script on the right (the data source and the specific layer), assigns it to the chosen name on the left of the symbol.

#lists one specific layer within a geospatial dataset with expanded information
nyc_trct<- st_read(dsn= "cb_2016_us_tract510_500k_nyc_wstats.gpkg", layer= "cb_2016_us_tract510_500k_nyc5") #read data (data set name, a layer of your choice)

Now, pulling the attributes from the data frame “nyc_trct” can vary in two ways, depending on how thoroughly you would like to comb through the data. First, you could pull out and examine just a short sample of data for a quick visual affirmation. The “head” function will pull out a sample of the FIRST six rows of the layer’s attribute table, with all of the fields listed out. The “tail” function will pull out a sample of the LAST six rows of the layer’s attribute table. Duplicate fields (columns) as well as statistics may appear as this layer has been joined with calculated race demographic statistics from a previous R code script. The tail function in particular is always a good quality check for any extra blank rows. Calculate the Location Quotient

How are we going to ensure that a group is actually represented in the geographic parcel, tract, or context? This is where the location quotient comes in. The location quotient situates and local geography with a regional geography - comparing each and thus contextualizing local patterns by calculating how more or less one group is represented or underrepresented. One may think that regionally, each census local tract is expected to look generally the same, but the social processes and contexts make them not the same. Surrounding census tracts influence the census tract. When dealing with a social problem it is not helpful to say that “there is this asian population here and this white population there”, it is better to say that these is a magnitude: “a 6x times higher asian population here than white population next to it”.

Before utilizing the fields in this dataset for calculations of the location quotient, using the function “summary” will calculate quick statistics (mean, median, quartile, etc.) for the defined context of each race population. One way to do this efficiently is by writing a loop. A “for” loop is a great tool for repetitive coding tasks by optimizing repeated functions among however many variables defined. First, a data value will be created called “raceGroups” will contain a combined “c” list of each race group. This “raceGroups” will appear in the environment tab under values. Then it will be indexed “i” into the “for” loop (the assignment of those variables to the index called “i” is user defined), that will “print” out the group names above the “summary” function statistics in the console space.

  print(summary(nyc_trct[[i]])) #print the summary stats of the race group from the data frame in the loop iteration
}

The location quotient manifests itself as a ratio index explicitly calculated below, following the initial equation comparing the ratio “part” to the “whole”. First, a new function titled “calculate lq” is created that is going to take the line of code right of the assignment operator “<-” and process it in the lines below. The “grp” is the local group of a census tract. The “ttl” is all of the population, or total population in the census tract. The “grp” is the numerator and the “ttl” is the denominator of the part. With those variables defined and appeared on the environment tab under functions, the local proportion in the numerator of the whole equation can be calculated by dividng the grp to the ttl and assigning that result to a new part called “lprp”, or the local proportion. Next, the regional proportion in the denominator of the whole equation can be calculated by dividng the sum of the grp and sum of the ttl, which is essentially first using the function “sum” to add up the population of the local group across all census tracts (grp) and the “sum” to add up the population of all of the population across all census tracts. The division of both of these sums is assigned to a new part called “rprp”, or the regional proportion. With both parts of the whole equation calculated and assigned, the “lprp” and “rprp” can be divided (part to part) to achieve the location quotient. The “return” function essentially closes out the created function (“calculate lq”).

"calculate_lq"<- function(grp, ttl){
  lprp<- grp/ttl #calculates the local proportion (part to the whole).
  rprp<- sum(grp)/sum(ttl) #calculates the regional proportion (part to the whole).
  lq<- lprp/rprp #calculates location quotient (part to part).
  return(lprp/rprp) #return the location quotient.
}

The result of this function calculation will be placed into the data table under the fields “WLQ, BLQ, etc.” The same is done for the other races.

Data Preparation for Exportation

With all of the statistics calculated, in the final step before the data is ready for use in GIS mapping program such as ArcPro and QGIS, there are a couple of formatting parameters to be addressed. This is a large amount of data that will be output from program to program, so it is always beneficial to compartmentalize it. To do this, a new data frame will be created using the assignment operator “<-” called “nyc_trct_df”, to convert the geographic data of “nyc_tract” to a simple data frame. This will ensure all function on the data can work. It can save data to link it with the layer when in the GIS because when you try to work on just the attribute table in the data frame there can be confusion in the program.

If you examine the attribute table, you will notice the extensively long strings of text in the “shape” field. This is too long and unnecessary–a lot of coordinates would be stored in the output. To remove the field, we can simply set it to be null. While more inconsistent bits and pieces of null values can negatively affect your data as a whole, if the entire field column is null than the variable will not be utilized in calculations at all and should not be a detriment. To do this, summon the field in the new data frame using the “$” sign and then assignment operate “<-” the values to be “NULL”.

#Converting the geographic dataframe to a standard dataframe
nyc_trct_df<- as.data.frame(nyc_trct)

#Remove the geometry by deleting the column. In the vignette, describe why this task is being done.
nyc_trct_df$Shape<- NULL

To ensure no inconsistencies in the data with pesky N/A’s (not available data), and inf’s (infinity), they will all be converted to 0. It is reasonable to change all of these to 0 based on the context that the census tracts with those values are those with no actual population residing within their bounds. With other datasets, context is important in determining how to deal with these values. To convert these to 0, the combined “c” group consisting of our new location quotient calculations pulled from the simple dataframe will be examined for their “is.na” values, or for those not available. Then, each of these groups with any NA data (that fall true under the condition that it “is.na”) will be reassigned from NA to 0. The same is done for the infinity values, except that the “is.na” condition is replaced with the “is.infinite” condition.

nyc_trct_df[c('WLQ', 'BLQ', 'ALQ', 'OLQ')][is.na(nyc_trct_df[c('WLQ', 'BLQ', 'ALQ', 'OLQ')])]<- 0
nyc_trct_df[is.na(nyc_trct_df$WLQ)]= 0
nyc_trct_df[is.na(nyc_trct_df$BLQ)]= 0
nyc_trct_df[is.na(nyc_trct_df$ALQ)]= 0
nyc_trct_df[is.na(nyc_trct_df$OLQ)]= 0

nyc_trct_df[is.infinite(nyc_trct_df$WLQ)]= 0
nyc_trct_df[is.infinite(nyc_trct_df$BLQ)]= 0
nyc_trct_df[is.infinite(nyc_trct_df$ALQ)]= 0
nyc_trct_df[is.infinite(nyc_trct_df$OLQ)]= 0

The last component before exportation would be to subset the demographic race group data using the “subset” function, and shrink the data down to only those fields needed, which are the fields that we created while calculating the location quotient, and then the “GEOID” (geographic identifier) field, which is a unique feature identifier, essential for the geographic linkage in mapping. Without it, the data would no longer be spatial data–just data. This sub-setting will be important in future steps on the GIS program as only the calculated location quotients will be necessary to merge onto the census tract geopackage “cb_2016_us_tract510_500k_nyc_wstats.gpkg”.

With our data now compartmentalized, it can now be exported as a new QGIS-native geopackage file (gpkg) called “nyc_trct_wlqs.gpkg” which contains a table called “lpq_data” for use in other programs or archiving through the “st_write” function.

lq_data<- subset(nyc_trct_df, select= c(GEOID, WLQ, BLQ, ALQ, OLQ))

st_write(lq_data, "nyc_trct_wlqs.gpkg", layer= "lq_data", delete_dsn = TRUE)
